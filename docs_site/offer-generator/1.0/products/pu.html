Pumping Unit (PU-only)

Module: pu. Form: /hpu/pumping-only → pu_costing.html. Status: ✅ Complete. Shares the helper, form shell and render path with the Heating & Pumping Unit; PU omits the heating (oil preheater) scope and prices the pump skid directly.

Inputs

Field Meaning Units / Default

Fuel / oil selection

No fuel dropdown. PU is for pre-heated heavy oils (LDO / LSHS) where heating is handled externally; the oil is represented by the Pumping Unit itself (no separate oil line). is_oil=True, is_dual=False are still set so the pumping scope renders (main.py:3258).

— (n/a)

Variant

hpu_variant — same field name as HPU; maps via PU_VARIANT_PREFIX (hpu_selector.py:33): Simplex→PUS, Duplex 1→PUD, Duplex 2→PUDD.

Simplex / Duplex 1 / Duplex 2. Default Duplex 1.

kW capacity

hpu_kw — unit kW, catalog key into pumping_unit_price.

3…48 kW. Default 9 kW.

Oil Flow Rate (LPH)

Catalog-driven, read-only — identical source/path to HPU (pumping_unit_price.flow_lph, GET /api/hpu/flow-lph). The pump skid is the same for both products (main.py:3131-3133).

Litres/hour. Auto.

Control mode

Fixed "manual" (main.py:3262).

"manual"

Quantity

qty — clamped to max(1, qty).

integer. Default 1.

Packaging & Forwarding % / Designing % / Negotiation % / Transport

Identical commercial inputs to HPU (pf_pct, design_pct, neg_pct, transport_amt). Same HpuQuoteRequest model.

%, %, %, ₹. Defaults 5 / 10 / 0 / 0.

Absent vs HPU

No heating/combustion inputs — no burner, no preheater, no combustion-air/blower, no fuel-CV or auto-ignition fields. For PU the heater is additionally absent from the product itself: scope and price exclude the electric Oil Preheater.

—

Outputs

  • Output template: PU_Offer_Template.docx (main.py:3176). The PU template reuses the same hpu_* context keys, so one context dict serves both (main.py:3231-3240).

  • Equipment label: "Pumping Unit"; item_product_type = "Pumping Unit". Model code = PUS|PUD|PUDD-{kW} (main.py:3172-3176).

  • Scope intro + bullets (pumping-only branch): force_pumping_only is True for PU (main.py:3261), so pumping_unit_block (quote_writer.py:319-337) emits heading _"Pumping Unit" with intro "To supply fuel oil to the above burner at the requisite pressure, we will supply a Pumping Unit type {variant}…" and bullets:

    • N pump(s) each with electric motor (2 for Duplex / standby pair, 1 for Simplex)

    • Duplex coarse + fine filters (cold/hot side)

    • Pressure regulating valve

    • Pressure gauge

      No preheater bullet and no temperature gauge — this is exactly how PU differs from HPU in scope.

  • Price schedule (Annexure III): single equipment line; same suppression of Vertical/Horizontal/Tundish scope and same vertical-slot total routing as HPU. Transport itemised on its own line; P&F / Designing / Negotiation folded into the unit price; TOTAL = all-inclusive grand_total.

  • Result payload identical shape to HPU with mode="pu"; filename {date}_{Customer}_PU-{kW}kW-{variant}.docx.

Calculation Flow

Shared with HPU except where noted. PU posts to POST /api/generate-pu-quote → generate_pumping_unit_offer(req, mode="pu") (main.py:3350-3356). Both products run the _same helper, form, template shell and render path; only the price source, model-code prefix, equipment label, filename infix and template file differ (selected by mode, main.py:3115-3122).

  1. Form → request. pu_costing.html posts the same payload to /api/generate-pu-quote (pu_costing.html:523); same HpuQuoteRequest.

  2. Flow-LPH lookup. Identical to HPU — pumping_unit_price.flow_lph by (kW, variant) (main.py:3134-3139). The pump skid is shared.

  3. PU selection + unit price — OMITS the HPU markup step. unit_price = pumping_unit_price.sell_price for (kW, variant), read directly (main.py:3162-3171). This price is already markup-included (rebuilt from hpu_master minus heater/thermostat × 1.8), so HPU_MARKUP is NOT applied here (contrast the HPU page, step 3). Mirrors GET /api/hpu/price?mode=pu (main.py:3096-3103). Errors out if no pumping_unit_price row.

  4. Commercial-adjustment formula — identical (main.py:3186-3192): _adj_total = unit_price * qty * (1 + (_pf+_des+_neg)/100) + _trn; negotiation ADDS; round to nearest ₹1000 when any adjustment set; offer_unit = _final_incl / qty.

  5. Build form_data — identical except item_product_type="Pumping Unit", equipment_label="Pumping Unit", and crucially force_pumping_only=True (set because filename_infix == "PU", main.py:3261), which drives the pumping-only scope branch.

  6. Render. Same calculate_quote → generate_quote_docx, but with PU_Offer_Template.docx (main.py:3315-3316); pumping_unit_block takes the _pumping-only branch (quote_writer.py:319), omitting the heater/preheater scope.

  7. Transport break-out — identical (main.py:3317-3321).

  8. Respond — identical shape, mode="pu" (main.py:3323-3336).

Shared vs omitted summary: shared = form layout, flow-LPH lookup, commercial-adjustment math, transport break-out, rounding, render/docx assembly path, price-schedule structure. Omitted for PU = the SUM(hpu_master) × 1.8 HPU pricing (replaced by direct pumping_unit_price.sell_price), the heating/preheater scope (suppressed via force_pumping_only=True), and the HPU template/label/prefix (swapped to PU equivalents).
